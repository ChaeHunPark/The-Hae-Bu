# 🔬 @Transactional 유무에 따른 데이터 정합성 파괴 실험

## 1. 개요
* **목표:** `@Transactional` 어노테이션이 없을 때 JPA의 변경 감지(Dirty Checking)가 작동하는지 확인하기
* **가설:** 트랜잭션이 없으면 영속성 컨텍스트가 유지되지 않아, 객체 값을 바꿔도 DB에 반영되지 않을 것.

---

## 2. 1차 시도 (실패 상황 재현)

* **상황:** 서비스 메서드에 `@Transactional`을 붙이지 않고 엔티티 수정 시도.
* **결과:** 로그에 `UPDATE` 쿼리가 아예 찍히지 않음. DB 값도 그대로임.
* **원인 분석:**
    * **범인:** `@Transactional` 누락.
    * **매커니즘:** 트랜잭션이 없으면 영속성 컨텍스트가 메서드 종료 시점에 `flush`를 호출하지 않음. 즉, JPA가 관리하는 '장부'에만 적어두고 실제 DB에는 제출하지 않은 상태로 끝난 것.

---

## 3. 2차 시도 (해결)

* **상황:** 메서드 위에 `@Transactional` 추가 후 다시 실행.
* **결과:** `UPDATE` 쿼리가 정상적으로 발생하며 DB 값이 수정됨.

---

## 4. 핵심 해부

### 💡 왜 @Transactional이 있어야 수정이 될까?
* **프록시의 판 깔아주기:** `@Transactional`이 있으면 스프링이 프록시 객체를 통해 트랜잭션을 시작함.
* **영속성 컨텍스트 유지:** 트랜잭션 범위 안에서 1차 캐시와 스냅샷이 활발하게 작동함.
* **더티 체킹(Dirty Checking):** 트랜잭션이 끝나는 시점에 프록시가 `commit`을 호출하고, 이때 JPA가 "처음 사진(스냅샷)이랑 지금이랑 다르네?"를 감지해서 자동으로 `UPDATE` 쿼리를 날려줌.

### ❓ 없으면 왜 안 될까?
* 트랜잭션이 없으면 영속성 컨텍스트가 데이터를 잠시 들고는 있지만, 그걸 DB에 반영(Flush)해야 할 '명분'이나 '시점'을 찾지 못함. 결국 메서드만 종료되고 변경 사항은 휘발됨.

---

## 5. 결론 및 느낀점
> 단순히 자바 객체의 필드 값을 바꾼다고 DB가 바뀌는 게 아니라는 점을 확실히 깨달음. JPA의 마법 같은 변경 감지는 반드시 `@Transactional`이라는 울타리 안에서만 안전하게 일어난다는 것을 눈으로 확인함.
