# 🔬 @Transactional 프록시 자기 호출(Self-Invocation)의 배신

## 1. 개요
* **목표:** 트랜잭션이 없는 메서드에서 트랜잭션이 있는 내부 메서드를 호출할 때, @Transactional이 정상 작동하는지 확인하기
* **가설:** 내부 메서드에 @Transactional이 붙어 있으니, 당연히 트랜잭션이 적용되어 데이터가 수정될 것.

---

## 2. 1차 시도 (예상 실패)

* **상황:** `트랜잭션_없는_메서드(A)`가 같은 클래스 내의 `트랜잭션_있는_메서드(B)`를 직접 호출함.
* **결과:** UPDATE 쿼리가 발생하지 않음 (10,000원 그대로 유지).
* **원인 분석:**
    * **범인:** 스프링의 **프록시(Proxy)** 메커니즘.
    * **매커니즘:** 스프링의 트랜잭션은 외부에서 프록시 객체를 통해 들어올 때만 가로채서 작동함. 하지만 클래스 내부에서 `this`를 통해 메서드를 호출하면 프록시를 거치지 않고 실제 객체의 로직을 직접 타기 때문에 @Transactional이 완전히 무시됨.

---

## 3. 2차 시도 (해결)

* **상황:** 호출부인 외부 메서드(A)에도 `@Transactional`을 추가함.
* **결과:** 트랜잭션이 정상 시작되어 내부 메서드(B)의 변경 사항도 DB에 반영됨. (성공)

---

## 4. 핵심 해부

### 💡 왜 내부 호출은 트랜잭션이 씹힐까?
* **대문만 지키는 프록시:** 스프링이 만든 가짜 대역(프록시)은 외부 호출만 감시함. 이미 대문을 통과해 들어온 내부에서 자기들끼리(this) 호출하는 건 프록시가 간섭할 수 없음.
* **AOP의 한계:** 스프링 AOP는 프록시 방식이라 발생하는 근본적인 한계임. (AspectJ를 쓰지 않는 이상 피할 수 없는 운명)

### ❓ 해결 방법은 무엇이 있을까?
* **가장 권장:** 메서드 A에도 `@Transactional`을 붙여서 트랜잭션 범위를 넓힘.
* **구조 분리:** 트랜잭션이 필요한 로직(B)을 아예 다른 서비스 클래스로 떼어내서 '남남'으로 만듦. 외부 호출 형태가 되면 프록시가 다시 일을 함.
* **자기 주입:** 조금 이상하지만 자기 자신을 `@Autowired`로 주입받아 `self.B()` 식으로 호출하면 프록시를 태울 수 있음.

---

## 5. 결론 및 느낀점
> 눈에 보이는 어노테이션이 전부가 아니라는 점을 뼈저리게 느낌. 스프링이 관리하는 'Bean'이 진짜 객체가 아니라 나를 감싼 '가짜 대역(프록시)'이라는 사실을 이해해야 트랜잭션의 배신을 막을 수 있음. 내부 호출 시에는 항상 이 프록시의 존재를 의식해야 함.
